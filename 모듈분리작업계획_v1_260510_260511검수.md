# 모듈 분리 작업 계획 v1 — 2026-05-10

> **분리의 목적은 분리 개발.** 매주·매달 새 모듈이 추가되는 상황 대비.
> 본 문서가 모듈 시스템의 헌법 SoT.

---

## 0. 의도 (Why)

### 0.1 핵심 의도
1. **분리 개발**: 한 사람이 한 모듈에 집중. 다른 모듈 영향 0.
2. **점진 추가**: 매주 신규 콘텐츠 (zone·route·시즌·이벤트) 모듈로 추가.
3. **독립 라이프사이클**: 추가·업데이트·제거가 다른 모듈에 영향 없음.
4. **콘텐츠 다양성**: 한 콘텐츠가 React·Unity-web·외부 라이브러리 등 다른 기술로 구현 가능 (각 모듈이 자기 기술 선택).

### 0.2 기획적 분리 (사용자 통찰)
- "주인공이 마이동을 데리고 진입한다" — 모듈 간 진입은 "들고 가는 데이터"가 명확.
- 주인공·마이동은 **광역 캐리어** (모든 모듈을 횡단)
- 각 모듈은 자기 영역의 **로컬 자원·로컬 데이터** 보유
- **세관 시스템 (Customs)**으로 로컬 ↔ 글로벌 자원 변환

### 0.3 안티 골 (이렇게 안 함)
- ❌ DLC 핫스왑 (런타임 모듈 추가·제거) — 빌드 시점만
- ❌ 모듈 간 직접 import (의존 강결합)
- ❌ 모듈 자율성 추구로 코어 변경 0 강제 → 비효율

## 1. 핵심 용어 정의

### 🏗️ 시스템 모듈 (System Module)
**재사용 가능 코어 엔진**. 한 번 만들고 여러 콘텐츠가 사용.

예:
- `vn-runner` — VN 시나리오 재생 엔진
- `gacha` — 가챠 메카닉 엔진 (확률·재시도·결과)
- `board` — 부루마블 메카닉 엔진 (주사위·이동·칸 도착)
- `customs` — 세관 변환 엔진 (로컬→글로벌)
- `cutscene-runner` — 컷신 시스템 엔진
- `codex-aggregator` — 도감 집계 엔진 (Phase 1.5+)
- `core-ui` — HUD·BottomNav·Layouts·Loading

특징:
- 데이터 적음, 로직·인터페이스 풍부
- export 기반 (다른 모듈이 사용)
- manifest `kind: 'system'`

### 📦 콘텐츠 모듈 (Content Module)
**실 데이터·UI·자산 인스턴스**. 시스템 모듈을 사용해 만들어진 구체.

예:
- `zone-garden` — 마이섬 3구역 (재배 미니게임)
- `route-neighbor` — 이웃섬 항로 (board 시스템 사용)
- `aidong-hwanggumeong` — 황금멍 캐릭터 (vn-runner·gacha 시스템 사용)
- `summer-watermelon-2026` — 2026 여름 수박섬 시즌 콘텐츠
- `attendance` — 출석부 (host에 부착)
- `merge-zone-14` — 14구역 머지 미니게임 (Unity-web 가능)

특징:
- 데이터·자산 풍부
- 시스템 모듈 require
- manifest `kind: 'content'`

### 🌐 광역 모듈 (Global Module)
**유저 세션 전체에 살아있는 광역 데이터·캐리어**. 모든 화면에서 read 가능.

예:
- `account` — 유저 계정 (uid·email·언어 설정)
- `host` — 유저캐릭터 (호스트명·코인·다이아·시간)
- `my-aidong` — 영입한 마이동 인스턴스 (현재·끊긴 모든 record)

특징:
- 항상 active (모듈 진입·이탈 무관)
- 다른 모듈이 read 가능 (write는 본인 모듈만)
- manifest `kind: 'global'`

### 🎨 자산 풀 (Asset Pool)
**리소스 (PNG·webp·MP3·JSON 시나리오 등)**. 모듈 외부에 별도 저장. **노출 보호 적용**.

예:
- `aidong/` — 본진 5명 PNG·표정·시그니처 (영입한 캐릭터만 클라 노출)
- `cutscenes/` — 컷신 시나리오 JSON (해금된 컷신만)
- `backgrounds/` — 배경 PNG (방문한 곳만)
- `audio/` — BGM·SFX

**노출 제어 2단계**:
- `public/` — 모두 fetch OK (오프닝·튜토리얼·UI 키트 등)
- `protected/` — backend 권한 검증 후 fetch (영입 캐릭터·해금 컷신·시즌 자산)

특징:
- 모듈에 종속 X
- backend가 user 권한 보고 signed URL·stream 제공 (Phase 1.5+)
- 1주차는 모두 public (단, frontend가 미영입 캐릭터 자산 fetch 안 함)

### 📊 공유 데이터 풀 (Shared Data Pool)
**모듈 종속 없는 공통 스태틱 메타 데이터**. CSV·JSON.

예:
- `aidong-master/` — 본진 5명 페르소나 파츠·시그니처·도감 줄·데뷔 보정
- `cutscene-catalog/` — 모든 컷신 정의 (호출 지점·트리거·시나리오 ref)
- `scenarios/` — 영입·정착·일기 JSON (현재 위치 그대로)
- `species/` — 5종 마스터 (강아지·고양이·곰돌이·햄스터·여우)

특징:
- 마스터 데이터 (game balance·캐릭터 정의 등 변경 빈도 낮음)
- 다수 모듈이 read (write 권한은 PM·기획만)
- 데이터 변경 시 다수 모듈 영향 → 신중

### 📊 5분류 용어 요약표 (v1.1)

| 용어 | 정의 | manifest kind | 예 |
|---|---|---|---|
| 🏗️ 시스템 모듈 | 재사용 엔진·인터페이스 | `system` | vn-runner·board·customs·cutscene-runner |
| 📦 콘텐츠 모듈 | 실 UI·인스턴스 | `content` | zone-garden·route-neighbor |
| 🌐 광역 모듈 | 항상 active 캐리어 | `global` | account·host·my-aidong |
| 🎨 자산 풀 | 리소스 (PNG·웹p·sound) | `asset-pool` (코드 X, 디렉토리만) | aidong·cutscenes·backgrounds·audio |
| 📊 공유 데이터 풀 | 스태틱 마스터 메타 | `shared-data` (코드 X, 디렉토리만) | aidong-master·cutscene-catalog·species |

→ 자산 풀·공유 데이터 풀은 **manifest 가지지 않음** (코드 모듈 X). 디렉토리·CSV·JSON 위주.

→ 본 문서 이후 **이 5 용어로 통일**.

## 2. 핵심 아키텍처 패턴 — 세관 (Customs)

### 2.1 의도 (사용자 통찰)
- 자원은 **로컬 자원** (한 모듈 안)과 **글로벌 자원** (모든 모듈 공유) 두 종류
- 모듈 경계 넘어가는 자원 이동은 **명시적 절차** = 세관
- "자원이동 최소화" — 세관을 거쳐야만 다른 모듈로 반출

### 2.2 예시 (사용자 제공)
```
[아이동개인섬-도토리볕] 모듈
  └ 채집자원: 도토리·열매·이끼 (로컬, 다른 모듈 모름)

[개인섬세관] (도토리볕 모듈 안 customs 시스템 인스턴스)
  └ 세관 테이블:
     · 도토리×3 → 항해보드 모듈의 '갑판 적재 식료품×1' 변환
     · 열매×5 → 글로벌 코인 ×10
     · 이끼×10 → 글로벌 다이아 ×1

[항해보드] 모듈
  └ 갑판 적재 자원: 식료품·물·연료 (로컬 자원)
  └ 갑판 → 마이섬 자원섬세관 → 마이섬 창고 (다음 변환)
```

### 2.3 customs 시스템 모듈 — 인터페이스
```typescript
// 시스템 모듈 customs/index.ts (export)
export interface CustomsTable {
  moduleId: string                      // 어느 콘텐츠 모듈의 세관인지
  conversions: CustomsConversion[]
}
export interface CustomsConversion {
  from: { resource: string; qty: number }   // 로컬 자원
  to: { resource: string; qty: number; targetModule?: string }  // 글로벌 또는 다른 모듈
  rate?: number                              // 변환 효율 (1.0 = 1:1)
  cooldownSec?: number                       // 변환 쿨다운
}

export function declareTable(table: CustomsTable): void
export function convert(moduleId: string, fromResource: string, qty: number): ConvertResult
```

### 2.4 콘텐츠 모듈이 자기 세관 등록
```typescript
// modules/zone-personal-acorn/index.ts
import { customs } from '@core/customs'
import { aiDongPersonalAcornCustoms } from './data/customs.csv.ts'

export function register() {
  customs.declareTable({
    moduleId: 'zone-personal-acorn',
    conversions: aiDongPersonalAcornCustoms,
  })
}
```

### 2.5 세관 테이블 SoT
- 각 모듈의 `data/customs.csv.ts` (또는 .csv) 파일
- 한국어 라벨·아이템 ID·수량·환율 모두 포함
- 모듈 제거 시 세관 자동 해제 (다른 모듈은 영향 없음 — 변환 path 사라질 뿐)

### 2.6 자원 이동 최소화 룰
- 모듈 안에서는 로컬 자원만 처리
- 다른 모듈로 자원 보내기 = 세관 통과 필수
- 글로벌 자원 (코인·다이아·gem)은 광역 모듈 (host)이 마스터링
- 세관 통과 시 항상 사용자 UI 확인 (자동 변환 X)

## 3. 핵심 아키텍처 패턴 — 컷신 (Cutscene)

### 3.1 의도 (사용자 통찰)
- 컷신은 게임 곳곳에 등장 (마이섬·하우스·항해·아이동개인섬·...)
- UI 단위는 비슷하지만 **참조 데이터·세팅이 다름**
- 따라서 **별도 시스템 모듈 = `cutscene-runner`** + **호출 지점별 로컬 세팅**

### 3.2 cutscene-runner 시스템 모듈
공통 엔진:
- 컷 시퀀스 재생 (페이드·텍스트·캐릭터)
- 트랜지션·이펙트
- 선택지·trigger

콘텐츠는 호출 측이 주입:
```typescript
cutsceneRunner.play({
  scenario: { /* JSON 시나리오 */ },
  context: { /* 호출 지점 데이터 */ },
  onComplete: () => { /* 호출 측 후속 */ },
})
```

### 3.3 호출 지점별 로컬 컷신
- `my-island/cutscenes/intro.json` — 마이섬 첫 진입
- `house/cutscenes/decorate-celebration.json` — 방 꾸미기 완료
- `board/cutscenes/storm.json` — 폭풍 칸 도착
- `zone-personal-{id}/cutscenes/first-visit.json` — 개인섬 첫 방문

### 3.4 오프닝·튜토리얼은 하드코딩 (예외)
- 오프닝 4단계 시퀀스·청소 4단계는 **강제 진행 → 일반 cutscene 시스템 사용 X**
- 직접 화면으로 (현재 OpeningScreen·CleaningLoopScreen 그대로)
- 이유: 정합성·UX 일관성 강제 필요

## 6. 모듈 manifest 표준
