## 한 줄 정의

아이동월드 웹 UI는 **모바일 게임을 좁게 옮긴 화면이 아니라, 아기자기한 분위기 안에서 숙소, 항해, 구역 생산, 도감, 인벤토리를 깊게 관리할 수 있는 웹형 생활 육성 UI**로 간다.

## 핵심 방향

- 웹은 넓은 화면과 여러 패널을 활용한다.
- 모바일 대응은 하되, 모든 화면을 모바일 폭에 맞춰 억지로 단순화하지 않는다.
- 숙소 중심 육성 허브와 메인 가변 구역을 Phase 1의 핵심 화면으로 둔다.
- 하단 메뉴는 유지할 수 있지만, 웹에서는 좌측/상단/보조 패널 구조도 적극 검토한다.
- route 분리는 유지보수 편의와 화면 상태 분리를 위한 것이며, 딥링크 제공 자체가 1차 목표는 아니다.
- 멀티탭과 멀티세션은 허용한다.
- QA는 멀티탭 사용 자체를 막지 않고, 복사 지급, 중복 소비, stale state 덮어쓰기 같은 버그를 막는 기준으로 잡는다.

## 시각 톤

목표:

- 아기자기함.
- 따뜻함.
- 장난감 같은 친근함.
- 웹 서비스처럼 읽히는 정돈감.
- 관리 게임처럼 반복 조작이 편한 밀도.

피할 것:

- 기본 MUI 데모처럼 보이는 밋밋한 카드 나열.
- 지나치게 무겁고 어두운 대시보드 톤.
- 모바일 앱을 확대만 한 듯한 큰 버튼과 빈 화면.
- 설명 문구가 너무 많은 튜토리얼식 화면.
- 모든 기능을 팝업 안에 밀어 넣는 구조.

## 레이아웃 원칙

웹 화면은 다음 계층을 기본 후보로 둔다.

```txt
상단: 현재 위치, 주요 자원, 계정/설정
좌측 또는 하단: 주요 이동 메뉴
중앙: 현재 핵심 장면
우측 또는 하단 패널: 선택 대상 상세, action, inventory, log
```

화면 크기별 기준:

| 화면 | 방향 |
|---|---|
| 넓은 desktop | 2~3 column, persistent side panel, 정보 밀도 높게 |
| tablet | 중앙 장면 + 접히는 상세 panel |
| mobile | 단일 column, bottom navigation, drawer/popup 활용 |

## 주요 화면 우선순위

1. 숙소 허브
2. 메인 화면과 가변 아이동 구역
3. 항해 만남/수집 화면
4. 통 인벤토리와 아이동별 도감 아이템 화면
5. 구역 배치 자동 생산 화면
6. 항구 입항/비우기 화면
7. 도감/업그레이드 화면
8. destination island/PixiJS/Rive 연출 POC

## 화면별 기준

### 숙소 허브

숙소는 Phase 1의 핵심 화면이다.

필수 요소:

- 방 카드 목록.
- 마당 또는 대기 Aidong 목록.
- 선택 Aidong 상세.
- 케어, 밥주기, 연습, 꾸미기, 업그레이드 진입.
- 방 수와 적극 육성 가능 수 표시.
- 항해 중, 구역 배치 중, 비활성 상태 badge.

웹에서는 방 목록과 선택 Aidong 상세를 동시에 보여주는 2 column 구조를 우선 검토한다.

### 메인 화면 / 가변 구역

메인 화면은 고정 구역과 아이동별 가변 구역이 늘어나는 허브다.

네비게이션 후보:

| 후보 | 설명 | 장점 | 단점 |
|---|---|---|---|
| 구역 grid | 기본 구역과 Aidong 구역을 카드/타일로 나열 | 이해하기 쉽다 | 구역이 많아지면 스크롤이 길다 |
| 지도형 배치 | 섬 지형 위에 구역 marker 배치 | 세계관이 산다 | 구현과 반응형이 어렵다 |
| 카테고리 tab | 기본 구역, Aidong 구역, 생산 구역으로 분류 | 관리가 쉽다 | 지도감이 약하다 |
| 즐겨찾기 dock | 자주 쓰는 구역을 고정 노출 | 반복 조작이 편하다 | 관리 UI가 필요하다 |

초기 권장:

- grid + category tab 조합.
- 이후 여유가 생기면 지도형 배치와 즐겨찾기 dock을 검토한다.

### 항해 화면

항해 화면은 복잡한 보드보다 발견 템포와 현재 상태를 명확히 보여준다.

필수 요소:

- 현재 항해 상태.
- 다음 행동.
- 발견 로그.
- 적재량.
- 아이동 만남 또는 보상 결과.
- 입항 권유.

주사위/칸 수/지정 이동은 `.cloud/61_voyage_board_simplification_2026-06-08.md`를 따른다.

### 항구 화면

항구는 출항과 입항/비우기의 화면이다.

필수 요소:

- 현재 배.
- 적재량.
- 이번 항해 화물.
- 출항 가능 여부.
- 비우기 action.
- 배 변경/업그레이드 후보.

배 적재량과 입항/비우기 기준은 `.cloud/62_ship_cargo_port_cycle_2026-06-08.md`를 따른다.

### 통 인벤토리

통 인벤토리는 단순 item list가 아니라 사용처를 보여줘야 한다.

필수 요소:

- 일반 재료.
- 장착 아이템.
- 꾸미기 재료.
- 항해 화물 입항 결과.
- 부족한 재료 안내.
- 어느 화면에서 쓰이는지 표시.

아이동별 도감 아이템은 통 인벤토리와 섞어 보여주더라도 소유권은 분리해 표시한다.

### 구역 배치 자동 생산

필수 요소:

- 구역별 배치 슬롯.
- 배치 후보 Aidong.
- 예상 생산량.
- 누적 생산량.
- 회수 버튼.
- 효율/비효율 badge.

반복 조작 화면이므로 action 버튼은 찾기 쉬워야 하고, 생산량 숫자는 작지만 선명해야 한다.

## 멀티탭 / 멀티세션 QA 기준

허용:

- 같은 계정을 여러 탭에서 여는 것.
- 한 탭에서 숙소, 다른 탭에서 항해를 보는 것.
- 새로고침 후 상태를 다시 hydrate하는 것.

막아야 하는 것:

- 같은 보상을 여러 탭에서 중복 수령.
- 같은 아이템을 여러 탭에서 중복 소비.
- 오래된 탭이 최신 상태를 덮어쓰기.
- 같은 Aidong을 여러 장소에 중복 배치.
- 항해 중인 Aidong을 숙소/구역에서 동시에 조작.

권장:

- backend action은 idempotency와 상태 조건 검증을 가진다.
- frontend는 stale state 가능성을 고려해 action 후 backend 응답을 병합한다.
- 중요한 action에는 최신 상태 재조회 또는 version/timestamp 비교 후보를 둔다.

## 라우트와 딥링크 기준

라우트 분리는 유지한다.

이유:

- 화면 책임 분리.
- Playwright smoke와 QA 경로 안정화.
- browser history와 refresh 대응.
- module route mount 유지.

하지만 Phase 1 목표는 딥링크 공유가 아니다.

따라서:

- `/island/lodge`, `/island/harbor`, `/voyage/board` 같은 route는 유지한다.
- 복잡한 modal 내부 상태까지 URL로 표현하지 않아도 된다.
- 인증/온보딩/항해 상태가 맞지 않으면 적절한 허브로 되돌린다.
- URL 직접 진입보다 앱 내부 흐름을 우선 검증한다.

## PixiJS / Rive 적용 위치

PixiJS와 Rive는 전체 UI 엔진이 아니라 연출 레이어다.

우선 후보:

- 항해 발견 연출.
- 보상 획득 연출.
- destination island POC.
- 숙소 케어 반응.
- Aidong idle/tap/reward reaction.
- 구역 생산 완료 연출.

React/MUI가 계속 담당할 것:

- 라우팅.
- HUD.
- 인벤토리.
- action 버튼.
- form.
- modal.
- 표와 목록.
- 상태 badge.

## UI 디자인 착수 전 체크리스트

- [ ] 숙소 허브의 desktop/tablet/mobile wireframe을 만든다.
- [ ] 메인 가변 구역의 grid/tab/map 후보 중 1차안을 정한다.
- [ ] 항해 화면의 보드/지정 이동 여부와 상관없는 공통 HUD를 정한다.
- [ ] 통 인벤토리와 아이동 도감 아이템 표시 방식을 분리한다.
- [ ] 구역 생산 화면의 배치 후보와 효율 표시 방식을 정한다.
- [ ] 항구 입항/비우기 화면의 적재량 표시를 정한다.
- [ ] 반복 action의 disabled/error/success 상태를 정의한다.
- [ ] Playwright smoke가 확인할 주요 route와 버튼을 정한다.

## 당장 하지 않을 것

- 모든 화면을 모바일 앱처럼 단일 column으로 고정.
- 모든 상태를 popup 하나로 처리.
- 딥링크 공유를 위해 복잡한 UI 내부 상태를 URL에 과도하게 넣기.
- PixiJS로 inventory, list, modal, form까지 만들기.
- 30칸 보드 확정 전제의 고급 항해 UI.
- 세관을 다시 핵심 하단 메뉴로 올리기.

## 후속 작업

3. 항구/항해 화면에서 보드와 적재량 HUD를 분리한다.
4. 통 인벤토리와 아이동 도감 아이템 UI를 별도 section으로 나눈다.
