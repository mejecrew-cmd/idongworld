이 문서는 항해, 배, 숙소, 세관, 모듈별 인벤토리 요구사항을 현재 구조와 대조하고 다음 구현 작업을 atomic하게 정리한다.

## 요구사항 요약

- 사용자는 내 섬을 가진다.
- 항해할 때 배를 여러 척 동시에 갖는 것은 아니지만 배 종류를 바꿀 수 있다.
- 배 종류는 선실 수, 갑판 슬롯, 적재량, 기타 능력치로 규정한다.
- 선실이나 갑판 slot에는 Aidong을 배치할 수 있다.
- 주사위를 굴려 다른 섬에 도착하면 해당 섬 모듈의 미션을 진행하고 module-specific resource를 얻는다.
- 다른 섬 모듈 자원을 배에 싣기 전에는 customs를 통과해야 한다.
- 다른 섬 모듈에서 획득한 아이템 중 전역 보관 대상은 customs를 통해 전역 보관소로 직접 이동할 수 있다.
- 배에서 숙소로 item/resource를 옮길 때도 customs를 통과해야 한다.
- 주사위로 도달하는 섬, 내 섬의 각 지역, 배, 숙소는 모두 개별 모듈 형식으로 구현한다.
- 개별 모듈은 각자의 인벤토리를 가진다.
- Aidong 소지/착용 아이템은 전역 인벤토리에 보관한다.
- 모듈별 specific item 목록은 해당 모듈에서 정의한다.

## 현재 구조 판정

### 만족하는 지점

- 마이섬은 `my-island` global module과 `myIslandStates` dedicated collection으로 존재한다.
- 내 섬 zone은 `zone-garden`, `zone-oasis`, `zone-memory`, `zone-mine` content module로 분리되어 있다.
- 항해 보드는 `route-neighbor` content module과 `routeNeighborStates` dedicated collection으로 분리되어 있다.
- 모듈별 backend state/API/repository/model spec 분리 원칙은 적용되어 있다.
- zone 모듈은 `zoneStates.localResources`에 module-local resource를 저장한다.
- `route-neighbor`는 `localResources`를 가지고 있고 customs resource adapter에 연결되어 있다.
- `ship`은 backend dedicated module과 frontend module package로 등록되어 있다.
- `shipStates`는 `shipTypeId`, `shipInventory`, `cabinAssignments`, `deckAssignments`, `cargo`, 호환용 `cabins`를 가진다.
- customs는 module resource와 host resource 사이 debit/credit, audit log, idempotency, rollback 기준을 가진다.
- host/global inventory는 `hostStates.inventory`와 frontend `userStore.inventory`로 존재한다.
- zone reward 숫자와 allowlist는 각 zone `balance.csv`에서 읽는 구조로 1차 분리되었다.

### 아직 부족한 지점

- 숙소는 화면(`/island/lodge`)은 있으나 정식 module package/backend storage가 아직 없었다.
- `ship.shipInventory -> lodge.lodgeInventory` 이동 rule은 아직 없다.
- route landing 결과가 “도착 섬 module local inventory -> ship inventory”로 이어지는 공식 flow가 아직 없다.
- 모듈별 `items.csv` item catalog 표준이 생겼고, 1차 대상 모듈에 catalog가 추가되었다.
- Aidong 소지/착용 item과 일반 host inventory를 구분하는 catalog/API 계약이 아직 없다.
- frontend 화면은 일부에서 아직 Zustand 통합 store를 직접 읽는 구조가 남아 있다.

## 구현 방향

- 서버는 하나로 유지한다.
- 구현 단위는 module package, manifest, backend model/spec/repository/service/API로 분리한다.
- `hostStates.inventory`는 Aidong 소지/착용 가능 아이템의 전역 보관소로 유지한다.
- module-specific resource와 item은 먼저 각 module inventory에 들어간다.
- module inventory에서 다른 module inventory 또는 host/global inventory로 이동할 때는 customs를 통과한다.
- 배는 `ship` module의 전용 상태를 권위로 삼는다.
- 숙소는 `lodge` module의 전용 상태를 권위로 삼는다.
- 배 종류, slot 구조, item catalog는 코드 하드코딩보다 CSV/config를 우선한다.
