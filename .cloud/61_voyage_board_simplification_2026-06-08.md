이 문서는 2026-06-05 기획 변경 회의 이후 항해 보드의 주사위, 칸 수, 이동 방식을 재논의하기 위한 결정 자료다.

## 한 줄 정의

항해 보드는 Phase 1에서 **아이동 만남, 재료 획득, 도감 아이템 획득을 가볍게 반복시키는 발견 장치**로 본다.

30칸 주사위 보드를 확정 밸런스로 삼지 않고, 첫 사용자 난이도와 반복 피로도를 기준으로 이동 방식을 다시 결정한다.

## 현재 전제

- 현재 구현은 `route-neighbor`가 항해 진행을 소유한다.
- `routeNeighborStates.currentRoute`, `boardPosition`, `landings`가 현재 항해 상태의 중심이다.
- 현재 화면은 주사위와 보드 위치를 전제로 움직인다.
- 하지만 2026-06-08 기준 주사위 채택 여부, 칸 수, 이동 방식은 P0 미결이다.
- 따라서 결정 전까지 30칸 보드의 밸런스, 연출, 칸별 보상 고도화는 멈춘다.

## 임시 권장안

월요일 재논의 전까지의 임시 권장안은 **8~9칸 축소 보드**다.

이유:

- 현재 `route-neighbor`의 보드 위치 저장 구조를 유지할 수 있다.
- 30칸의 피로도를 줄일 수 있다.
- 주사위를 완전히 버리지 않고 테스트할 수 있다.
- 지정 이동으로 갈 경우에도 작은 보드 구조를 중간 단계로 재사용할 수 있다.

단, 이 권장안은 확정이 아니다.

주사위를 유지할지, 칸 수를 줄일지, 지정 이동으로 갈지는 P0 미결로 유지한다.

## 보드 이벤트 재정의

보드 칸은 더 이상 destination island local resource와 세관 gate를 기본 전제로 삼지 않는다.

Phase 1 보드 이벤트 후보:

- `aidong-encounter`: 아이동 만남 후보.
- `material-gain`: 통 인벤토리 재료 획득.
- `codex-item-gain`: 특정 Aidong 도감 아이템 획득.
- `mini-event`: 짧은 스토리, 선택지, 컷신.
- `rest`: 항해 중 휴식 또는 보상 없음.
- `return-hint`: 적재량 또는 진행도에 따라 입항 권유.

보상 권위:

- 일반 재료는 `hostStates.inventory`에 지급한다.
- 아이동 도감 아이템은 `myAidongStates.aidongCodexItems`에 지급한다.
- 아이동 만남은 `myAidongStates`와 `myIslandStates` 상태 전이로 처리한다.
- 세관은 이 흐름의 필수 gate가 아니다.

## Feature Flag 후보

결정 전까지 보드 메카닉을 교체 가능하게 하려면 다음 feature flag 후보를 둔다.

```text
VITE_VOYAGE_BOARD_MODE=dice30
VITE_VOYAGE_BOARD_MODE=dice9
VITE_VOYAGE_BOARD_MODE=direct
VITE_VOYAGE_BOARD_MODE=hybrid
```

backend도 같은 의미의 env 또는 config 후보를 둘 수 있다.

```text
VOYAGE_BOARD_MODE=dice30
VOYAGE_BOARD_MODE=dice9
VOYAGE_BOARD_MODE=direct
VOYAGE_BOARD_MODE=hybrid
```

초기에는 env를 바로 구현하지 않아도 된다.

다만 route service와 frontend board scene은 board size, roll rule, landing generator를 config로 뺄 수 있는 구조를 의식하고 작업한다.

## 코드 작업 금지선

결정 전까지 하지 않는다.

- 30칸 보드 전용 밸런스 고도화.
- 30칸 전용 asset과 연출 대량 추가.
- 30칸 칸별 보상 테이블 확정.
- 주사위 확정 전제의 tutorial 문구 작성.
- destination island 세관 gate와 항해 보드를 다시 강하게 묶는 작업.

허용한다.

- 현재 항해 상태 저장 안정화.
- 보드 진입/복귀/새로고침 안정화.
- landing event payload를 `aidong-encounter`, `material-gain`, `codex-item-gain` 중심으로 정리하는 작업.
- 보드 크기와 이동 방식을 config로 바꾸기 쉬운 리팩토링.
- feature flag 후보 문서화.

## 재논의 질문

월요일 재논의에서 결정해야 할 질문:

1. 항해에 주사위가 꼭 필요한가.
2. 보드가 필요하다면 몇 칸이 적당한가.
3. 첫 항해에서 아이동을 만나는 데 몇 번의 조작이 필요한가.
4. 사용자가 원하는 아이동이나 재료를 어느 정도 목표 지향적으로 찾을 수 있어야 하는가.
5. 배 적재량은 보드 진행 중 언제 압력으로 작동해야 하는가.
6. 항해 실패, 빈 칸, 보상 없음 이벤트를 허용할 것인가.
7. 항해가 미니게임인지, 보상 선택 화면인지, 발견 로그인지 어디에 가까워야 하는가.

## 결정 후 작업 후보

주사위 유지 시:

- board size를 config화한다.
- `roll` 결과와 landing generator를 분리한다.
- 튜토리얼에서 주사위 의미를 설명한다.

8~9칸 축소 시:

- `route-neighbor` balance/config에 board size를 둔다.
- 현재 board position 저장은 유지한다.
- 화면은 한눈에 전체 보드가 보이도록 단순화한다.

지정 이동 시:

- `roll` action을 `choose-destination` 또는 `explore` action으로 대체한다.
- board position은 진행 단계 또는 탐색 depth로 낮춘다.
- UI는 목적지 카드, 보상 후보, 위험도 표시 중심으로 바꾼다.

하이브리드 시:

- destination 선택과 짧은 랜덤 이벤트를 분리한다.
- 사용자는 방향 또는 목적지를 고르고, backend는 해당 목적지 안에서 event를 생성한다.
- 보드는 큰 지도보다 짧은 탐색 경로 표현으로 둔다.

## 완료 기준

- 주사위 채택 여부를 P0 미결로 유지한다.
- 30칸/축소/지정 이동/하이브리드 선택지를 비교했다.
- 첫 사용자 난이도와 항해 피로도 기준의 UX 검토표를 만들었다.
- 결정 전 30칸 보드를 더 고도화하지 않는 금지선을 명시했다.
- feature flag 또는 config 전환 후보를 기록했다.
