# 다음 작업 메모 2026-06-03

> 2026-06-12 기준 이 문서는 과거 작업 이력으로 보존한다. 최신 실행 보드는 `.cloud/72_next_work_2026-06-12_week1_a.md`를 따른다.

이 문서는 2026-06-01 작업 이후 다음 개발 순서를 정리한다. 현재까지 backend 분리, MongoDB local 저장, 모듈별 action API, customs, ship/lodge/route 기본 구조, opening 반복 방지, PixiJS/Rive 부분 도입 방향까지 정리된 상태를 출발점으로 둔다.

## 현재 결정된 방향

- backend는 단일 서버를 유지하되 account, host, module, customs API를 분리한다.
- module-local state는 전용 document와 전용 action API를 우선 사용한다.
- 모듈 간 자원 이동은 customs가 담당한다.
- 항해 도중 도착하는 섬은 `destination-island` worldScope를 가진 독립 모듈로 다룬다.
- destination island는 Aidong이 직접 걸어 다니는 화면이 아니라, 고정맵의 hotspot을 클릭하며 상호작용하는 탐험 화면이다.
- PixiJS는 destination island의 배경 layer, particle, light, fog, water, transition 같은 화려한 scene effect에 부분 도입한다.
- Rive는 Aidong 반응, hotspot 반응, 보상, 세관/미션 UI 애니메이션의 1차 선택지로 둔다.
- Spine/Live2D는 작업량이 크므로 현재 단계에서는 보류한다.
- public asset은 일반 CDN/public URL로 제공하고, protected/해금형 asset은 signed URL을 검토한다.

7. Asset Access 정책 정리 - 완료 2026-06-05
   - public asset과 protected asset을 구분한다.
   - DB에는 signed URL을 저장하지 않고 `assetId`, `assetKey`, `version`, `visibility`만 저장한다.
   - public asset은 일반 public path 또는 CDN URL을 쓴다.
   - protected/해금형 asset은 backend 권한 확인 후 signed URL을 발급하는 구조로 설계한다.
   - signed URL은 userStore에 장기 저장하지 않는다.
   - `.cloud/46_asset_access_policy.md`를 추가했다.
   - DB/CSV/module state에는 URL이 아니라 `assetId`, `assetKey`, `category`, `visibility`, `version`만 저장한다는 기준을 확정했다.
   - public asset은 URL helper가 즉시 public URL로 해석한다.
   - protected asset은 URL을 직접 만들지 않고 signed URL resolve 대상으로 둔다.
   - core `asset-helper`에 `AssetRef`, `AssetResolvePlan`, `assetCacheKey`, `resolveAssetRef`를 추가했다.
   - signed URL 발급 API skeleton은 8번 작업으로 남긴다.
   - `pnpm --filter @idongworld/core test -- asset-helper.test.ts`로 검증한다.
