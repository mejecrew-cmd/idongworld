# 🛠 툴 리스트 v1

> 코드는 모듈로, **데이터/콘텐츠는 툴로** — 작성 양식·생산 절차·검증 파이프라인을 도구 단위로 캡슐화.

---

## 0. 의도 (왜 이 문서)

- 데이터 종류마다 **양식·생산자·검증 방식**이 다른데 흩어져 있으면 일관성이 무너짐.
- 도구를 한 곳에 등록 → 본진 5명 보이스·영감원 검수·페이스를 매주 점검 가능.
- 코드 모듈(모듈분리작업계획) 과 별도로 **콘텐츠 흐름** 의 SoT 역할.
- AI 생성(twin)을 사용할 경우 **검수 게이트**가 필수 — 본 문서가 그 룰을 정의.

## 2. 도구별 사양

### 2.1 툴_컷신제작 (메인)

> **단일 진실 소스**: 모든 시나리오·컷신·일기 텍스트는 본 도구를 거쳐서만 산출.

#### 2.1.1 처리 대상 (양식 2종)

| 양식 | 용도 | 스키마 위치 | 현황 |
|---|---|---|---|
| **VN Scenario** | recruit·settle·cutscene·dynamic_reveal·dialogue | `@idongworld/vn-runner/types` (Scenario·Scene·SceneChoice) | 11편 작성 |
| **Diary Fragment** | 일기 (5 시간대 슬롯) | 기획/시나리오/diary_*.json (fragments + slots) | 5편 → 100편 목표 |

→ **두 양식의 정합**:
- Diary 도 vn-runner 가 재생할 수 있게 어댑터 필요 — `vn-runner` 분리 시점에 검토. 현재는 DiaryModal 자체 렌더.
- 신규 양식 도입 시 본 §2.1.1 표 갱신 + scheme TS 타입 동기.

#### 2.1.2 생산 정책 (3단계)

```
[1] 아웃라인  →  [2] 표 채우기 (작가)  →  [3] 검수 게이트  →  [4] JSON 변환 (PM)
   ↑ PM         ↑ 작가 / Twin 보조      ↑ 도구 자동 + 사람    ↑
```

| 단계 | 입력 | 산출 | 책임 |
|---|---|---|---|
| **1. 아웃라인** | 캐릭터 ID·시점·목적·핵심 비트 (3~5줄) | outline 메타 표 | PM |
| **2. 표 채우기** | 작가가 outline 기반 [작성테이블](../../기획/작가/시나리오_작성테이블_v1_260510_260511검수.md) 양식 채움 | _초안_*.md | 작가 (Twin 보조 가능) |
| **3. 검수** | 스키마·길이·영감원 grep·보이스 일관성 | 통과/반려 + 코멘트 | 도구 자동 + PM 최종 |
| **4. JSON 변환** | 통과한 표 → scenario.json | scenario.json (스키마 적합) | PM |

→ **작가 작업 룰·보이스·금기**: [작가_시나리오지침서](../../기획/작가/작가_시나리오지침서_v1_260510_260511검수.md) 가 단일 SoT.

#### 2.1.3 Twin 사용 정책

**Twin = 페르소나 DNA 기반 LLM 생성** (메모리: [아이돌 페르소나 라이브러리 스킬 v1.0]).

| 항목 | 정책 |
|---|---|
| 사용 가부 | ✅ **권장** (일기 100편 등 대량 생산은 사실상 필수) |
| 입력 | 캐릭터 페르소나 #(56·75·24·7·19) + outline + 이전 day 컨텍스트 |
| 출력 검수 | **3중 게이트** 필수 — ① 스키마 ② 영감원 grep 0 ③ 사람 voice 점검 |
| 1편 자동승인 금지 | 첫 편(day 1)·시그니처·데뷔는 사람 작성 락 |
| Twin 미사용 시 | 작가 1인이 캐릭터별 보이스 일관성 책임 |

#### 2.1.4 일기 100편 생산 계획

**목표**: 5명 × 20일 = 100편 (1명당 20 day 슬롯).

| 단계 | 분량 | 시점 | 비고 |
|---|---|---|---|
| **현재** | 5편 (5명 × day1) | 작성 완료 | 사람 작성·voice 락 기준 |
| **MVP** | 25편 (5명 × day1~5) | Phase 1 베타 (5/24까지) | day1 사람·day2~5 Twin + 검수 |
| **POC 출시** | 60편 (5명 × day1~12) | Phase 1 POC (6/23) | 출시 1주 안 페이스 (1.7편/일) |
| **운영** | 100편 (5명 × day1~20) | Phase 2 (~7/31) | 시즌 콘텐츠로 통합 |

**day 분기 규칙**:
- day 1~5: 정착·일상 적응 (잔잔)
- day 6~10: 친밀도 +50 도달 시 해금되는 깊은 일기
- day 11~15: 마이섬 구역 해금 연동 (정원·창고·바다·...)
- day 16~20: 시즌 이벤트·생일·기념일 (시즌 모듈 콘텐츠)

#### 2.1.5 도구 위치·실행

```
v1_260509/tools/cutscene/                ← 생성 (Phase 1.5+)
├─ schema/                               ← scenario·diary JSON Schema
│  ├─ scenario.schema.json
│  └─ diary.schema.json
├─ templates/                            ← outline·twin 프롬프트
│  ├─ outline_recruit.md
│  ├─ outline_diary.md
│  └─ twin_prompt_diary.md
├─ scripts/
│  ├─ validate.ts                        ← 스키마 + 길이 + 영감원 grep
│  ├─ twinDraft.ts                       ← (Phase 1.5) Twin LLM 호출
│  └─ exportFromOutline.ts               ← outline → scenario.json (수동 작성용 빈 틀)
└─ README.md
```

**실행 명령** (계획):
```bash
pnpm tool:cutscene:validate <scenario.json>     # 검증
pnpm tool:cutscene:draft diary 황금멍 day7      # outline.md 생성 → 사람 또는 Twin
pnpm tool:cutscene:validate-all                 # 전체 일괄 검증
```

#### 2.1.6 검수 게이트 (자동)

| 검사 | 통과 조건 | 실패 시 |
|---|---|---|
| **스키마** | JSON Schema 검증 통과 | 반려 |
| **길이** | diary fragment characterCount ±20% / scene line ≤200자 | 경고 + PM 검토 |
| **영감원** | `정국·태민·창빈·레오·카이` grep **0 매치** | 즉시 반려 |
| **시그니처 일관성** | 캐릭터 시그니처 표현(Word) 출현 ≥1 | 경고 |
| **금칙어** | 비속어·실명·외부 IP | 즉시 반려 |

→ 게이트 통과 후 PM 사람 voice 점검은 별도 (도구 자동화 X).

## 4. 표준 입출력 형식

### 4.1 시나리오 JSON (VN Runner)
- 스키마: `packages/modules/vn-runner/src/types.ts` (Scenario·Scene)
- 위치: `기획/시나리오/{type}_{scenarioId}.json`
- 명명 규칙: `cutscene_*` · `recruit_*` · `settle_*` · `dynamic_reveal_*`

### 4.2 일기 JSON (Diary Fragments)
- 스키마: `기획/시나리오/diary_*.json` (현 구조 — 추후 schema TS 타입화)
- 필드: `id·version·characterId·personaId·day·layer·metadata·fragments{wake/morning/noon/evening/night}`
- 위치: `기획/시나리오/diary_{character}_day{N}.json`
- characterCount 필드: 길이 검증용 자가 보고

### 4.3 마스터 CSV (예정)
- 위치: `shared-data/{aidong-master,species,schedule}/*.csv`
- 인코딩: UTF-8 (BOM 없음)
- 헤더: 첫 행 = 컬럼명·snake_case
- 라인 종결: LF (\n)

### 4.4 모듈 manifest TS
- 스키마: `@idongworld/core/manifests` (defineManifest)
- 위치: `packages/modules/{id}/manifest.ts`
- 검증: `registry.validate()` 부팅 시점
