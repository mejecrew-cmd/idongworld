# 데이터 연결 다이어그램 — DB·JSON·CSV·기획 SoT 매핑

> 게임 데이터의 출처(SoT) → 코드 → 화면까지 전 흐름을 한 곳에 정리.

## 0. 전체 데이터 흐름

```
[기획 SoT (모듈·매뉴얼·CSV)]
        ↓
[캐릭터엔진 (a~e 시트)]
        ↓
[frontend data/* (TS 정적 데이터)]
        ↓
[stores/userStore (Zustand state)]   ←→   [backend repositories (users/module states)]
        ↓                                          ↓
[components/screens (UI)]                  [routes/* (API)]
        ↓
[와이어프레임/assets-dummy (PNG)]  ← Phase 2 실 PNG로 교체
```

## 2. 데이터 흐름 예시 — "황금멍 영입"

```
유저 클릭 "함께 갑니다" (recruit_hwanggumeong scene_05_settle)
  ↓
VNPlayer.tsx → choices[0].trigger 처리
  ↓
lib/vnTrigger.ts handleTriggers([
  "unlock_aidong:황금멍",      → store.recruitAidong('황금멍')
  "unlock_diary:황금멍_day1",  → store.unlockDiary(...)
  "navigate:settle_hwanggumeong" → 다음 시나리오
])
  ↓
my-aidong facade/action:
  - frontend store에 즉시 반영
  - backend action API 호출 준비
  ↓
api.recruitAidong(uid, '황금멍')
  ↓
POST /api/modules/my-aidong/recruit
  ↓
my-aidong service validation
  ↓
myAidongStates repository/model 영속
  ↓
응답 state를 frontend facade가 병합
  ↓
host/codex/diary 등 다른 소유권 변경이 필요하면 각 action API 또는 customs 경계를 통과
```

- 모델/collection: users · hostStates · myAidongStates · myIslandStates · codexStates · routeNeighborStates · shipStates · zoneStates · moduleStates · customsLogs
- frontend persist + split state/module API sync 병행

## Backend Persistence Update 2026-05-29

- 기존 Zustand -> backend 전체 state sync 다이어그램은 폐기됐다. 현재 흐름은 account, host, module, customs, action API 기준으로 설명한다.
- 현재 백엔드 영속화는 소유권 기준으로 분리되어 있다. account/user, host, 모듈 지역 상태, 범용 module state, ship/zone/route state, customs audit log가 분리 대상이다.
- 신규 모듈 데이터 흐름은 frontend module action -> module API -> module service -> module repository/model 순서로 설계한다.
- 모듈 간 자원 이동은 frontend customs request -> customs API -> resource adapter -> module/host repository -> customs log 순서로 처리한다.
- MongoDB 개발은 로컬 우선으로 진행한다. Atlas는 현재 사용하지 않고, 추후 사용할 수도 있는 선택지로 보류한다. 사용 확정과 credential 준비 뒤에만 readiness를 실행한다.
