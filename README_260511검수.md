## 상태 (Zustand)

User store (`stores/userStore.ts`) — localStorage 영속 + facade 우선 접근:
- Auth: `firebaseUid` `nickname` `isGuest`
- 자원: `coins` `diamonds` `gems`
- 진행: `onboardingComplete` `hostName`
- 캐릭터: `recruitedAidongs[]` `affinities{}` `firstGachaCandidate` `firstGachaAttempts`
- 마이섬: `unlockedZones[]` `unlockedDiaries[]` `unlockedCodexEntries[]` `codexFullyRegistered[]` `inventory{}`

Actions: `loginGuest` · `recruitAidong` · `addAffinity` · `completeOnboarding` · `unlockZone` · `unlockDiary` · `unlockCodexSlot` · `fullyRegisterCodex`

신규 frontend runtime 코드는 `useUserStore`를 직접 import하지 않고 `packages/frontend/src/lib/storeFacades.ts`의 facade를 우선 사용한다. backend 동기화는 전체 state patch가 아니라 account, host, module, customs, action API 경계를 사용한다.

## Backend Persistence Update 2026-05-29

- 백엔드 저장소 분리 Phase 1은 완료 상태다. 서버 프로세스는 하나의 Express 서버를 유지하되, 저장 책임은 repository, model, module route 경계로 나뉜다.
- MongoDB 개발 정책은 로컬 우선이다. Atlas 사용 여부는 아직 미정이며, 실제 전환은 결정과 credential 준비 후 `MONGO_URI`와 배포 secret 교체로 처리한다.
- 현재 백엔드 영속화 범위는 `users`, `hostStates`, 모듈별 전용 state collection, 범용 `moduleStates`, `customsLogs`를 포함한다.
- 신규 모듈 백엔드 작업은 모듈 전용 API와 repository를 사용한다.
- `/api/state` 전체 user state 동기화 경로는 제거됐다. 신규 기능은 account, host, module, customs, 명시적 action API에 붙인다.
- 세관(customs)은 모듈 간 자원 이동의 경계다. 한 모듈이 다른 모듈 document를 직접 수정하지 않는다.
- 현재 기본 검증은 `pnpm check:state-route-static`, `pnpm check:module-actions-smoke`, 필요 시 `pnpm check:frontend:state-route-runtime`을 사용한다.
