# 🔎 중간 작업 검수 절차

> **언제 사용**: 모듈 분리 단위 작업 1개 완료 직후 + 매주 금요일 정기 점검 + Compact 직전.
> **목적**: 코드·문서·자산이 계획(모듈분리작업계획 v1.x)과 일치하는지 빠르게 확인.

---

## 0. 5분 요약

```
검수는 6단계:
  A. 워크스페이스 부팅           — pnpm install·typecheck
  B. 정합성 grep                 — 구 경로·미삭제 잔재
  C. 주석 헤더 점검              — 📁 마커·beginner 안내
  D. 모듈 manifest 정합          — exports 표면·requires 무결
  E. 문서 cross-ref              — SoT 문서끼리 동기화
  F. PM 결정 일관성              — 본진 5명·5분류·자산 보호

총 소요: 10~15분 (자동 grep + 수동 검토).
```

## 1. 단계별 절차

### A. 워크스페이스 부팅 (필수 — 자동)

```bash
cd v1_260509

# 1) 의존성 동기화
pnpm install        # → "Already up to date" or 신규 패키지 install

# 2) 전체 타입 검사
pnpm typecheck      # → 모든 패키지 (core·vn-runner·frontend·backend·...) Pass

# 3) (선택) 빠른 dev 부팅 검증 — UI 변경 시
pnpm dev            # FE 5173 + BE 4000 띄우고 한 사이클 클릭 후 Ctrl+C
```

**Pass 조건**: typecheck 모두 `Done`. 한 패키지라도 실패면 그 자리에서 중단·수정.

---

### B. 정합성 grep — 구 경로·잔재 검출 (필수 — 자동)

모듈 분리 후 자주 남는 stale 참조를 한 번에 잡음.

```bash
cd v1_260509/packages

# B-1. 삭제된 구 lib/* 경로가 어디서 import 되고 있는지
grep -rn "lib/vnTypes\|lib/vnTemplate\|lib/vnTrigger\|components/VNPlayer" \
  --include="*.ts" --include="*.tsx"

# B-2. 분리된 모듈을 frontend 외부에서 직접 깊게 import 하는지 (헌법 위배)
#      허용: '@idongworld/vn-runner' / 차단: '@idongworld/vn-runner/src/internal'
grep -rn "@idongworld/[a-z-]*/src/" --include="*.ts" --include="*.tsx"

# B-3. 영감원 식별어 0 매치 강제 (본진 5명 §1.1)
grep -rin "정국\|태민\|창빈\|레오\|카이\|jungkook\|taemin\|changbin\|leo\|kai" \
  --include="*.ts" --include="*.tsx" --include="*.json" --include="*.md"
```

**Pass 조건**:
- B-1, B-2: **0 hit**
- B-3: 의도적인 SoT 문서(본진5명선정.md)·grep 자기 참조 외 0 hit

---

### C. 주석 헤더 (📁 마커) 점검 (필수 — 자동)

모든 신규/이동된 TS·TSX 파일은 `📁 파일명 — 한 줄 요약` 헤더 필수.

```bash
cd v1_260509

# C-1. Phase 작업 폴더 내 헤더 누락 검출 (예: vn-runner)
for f in $(find packages/modules/vn-runner/src packages/core/src \
              packages/frontend/src/lib -name "*.ts" -o -name "*.tsx" 2>/dev/null); do
  head -3 "$f" | grep -q "📁" || echo "❌ HEADER 없음: $f"
done

# C-2. import만 있고 코드는 placeholder인 파일 찾기 (TODO 누적 검출)
grep -rln "TODO\|FIXME" packages/ --include="*.ts" --include="*.tsx" | head -20
```

**Pass 조건**:
- C-1: 모든 파일 헤더 보유 (출력 비어 있음)
- C-2: TODO 개수 추적 — 늘어나기만 하면 일주일 단위로 청산 일정 잡기

---

### D. 모듈 manifest 정합 (선택 — 수동 검토 + 향후 자동화)

각 모듈의 `manifest.ts` 와 실제 `index.ts` exports 가 일치하는지.

```bash
# D-1. manifest 파일 목록
find packages/modules -name "manifest.ts" -maxdepth 3

# D-2. 해당 모듈 index.ts 의 export 와 manifest.exports 비교
#      (현재는 수동 — Phase 1.5에 tools/checkManifests.ts 자동화 예정)
cat packages/modules/vn-runner/manifest.ts
cat packages/modules/vn-runner/src/index.ts
```

**Pass 조건**:
- `manifest.exports` ⊂ 실 export 표면
- `manifest.requires` 의 모든 ID 가 다른 모듈에 등록 (registry.validate)
- `kind` 가 폴더 위치와 일치 (system 모듈은 `packages/modules/`, 광역은 별도 합의 위치)

### E. 문서 cross-ref (필수 — 수동 + grep)

SoT 문서끼리 어긋나면 다음 세션에서 혼선이 큼.

```bash
cd v1_260509

# E-1. 핵심 SoT 문서 변경 시 changelog 동기 확인
grep -l "Changelog" *.md 기획/**/*.md 2>/dev/null | head

# E-2. 문서 본문에 적힌 파일 경로가 실재하는지 (간이 sample)
#      예: 모듈분리작업계획에서 packages/core/src/manifests.ts 가 실제 있는지
grep -oE "packages/[a-zA-Z0-9_/-]+\.tsx?" 모듈분리작업계획_v1_260510.md | sort -u | \
  while read p; do [ -f "$p" ] || echo "❌ 미존재 경로: $p"; done

# E-3. COMPACT 인수인계 § 진행 상황 표가 최신인지 (수동)
#       → §4 우선순위 작업 5건의 [ ]/[x] 갱신 필요
```

**Pass 조건**:
- 본문 경로가 모두 실재
- 핵심 변경 시 모든 참조 문서의 changelog 갱신 (모듈분리작업계획 / COMPACT 인수인계 / 개발계획)
