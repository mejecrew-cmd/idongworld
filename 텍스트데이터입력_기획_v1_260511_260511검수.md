# 📝 텍스트 데이터 입력 기획 v1

> 모든 게임 내 텍스트(i18n·시나리오·일기·UI 라벨)는 **CSV/JSON 양방향 변환** 표준에 따른다.
> 작가는 Excel·Google Sheets·CSV 편집기를 자유롭게 쓰고, PM 이 검수·import → 모듈 JSON 자동 갱신.

---

## 0. 30초 요약

```
[작가/번역가]                [PM]                       [코드]
─────────────                ────                       ────
i18n_export.csv 편집  →   pnpm i18n:check              모듈 i18n/{ko,en}.json
(Excel/Google Sheets)     pnpm i18n:import              ↓
                          git commit                   화면 t('ns:key') 즉시 반영
```

## 2. CSV 포맷 사양 (RFC 4180)

### 2.1 컬럼 구조 — 단일 통합 CSV (전체 모듈 × 언어)

```csv
module,key,ko,en,description,status
zone-garden,title,🌱 성장의 정원,🌱 Growth Garden,zone 진입 헤더,locked
zone-garden,subtitle,"씨앗을 심고 30초, 기다리면 식물이 자라요.","Plant a seed, wait 30s, and it grows.",툴팁,locked
common,welcome,"안녕, ""호스트""!","Hello, ""host""!",대화 인용 예,review
```

**6 컬럼 표준**:
| 컬럼 | 타입 | 설명 |
|---|---|---|
| `module` | string | 모듈 ID (예: `zone-garden`·`my-island`·`common`) |
| `key` | string | i18n 키 (예: `title`·`sample.greeting`) — `.`구분 OK |
| `ko` | string | 한국어 |
| `en` | string | 영어 |
| `description` | string | (선택) 작가·번역가 메모 |
| `status` | enum | `draft` / `review` / `locked` |

### 2.2 인코딩·구분자
- **인코딩**: UTF-8 (BOM 없음)
- **구분자**: `,` (콤마)
- **줄 종결**: `LF (\n)` — Windows 편집기 시 `\r\n` 자동 변환
- **첫 행**: 헤더 필수
- **빈 줄**: 무시
- **주석 줄**: `#` 으로 시작 (CSV 파서가 별도 처리)

### 2.3 특수 문자 처리 (RFC 4180 준수)

| 케이스 | 처리 |
|---|---|
| 본문에 콤마 `,` | 전체 필드를 `"..."` 인용 |
| 본문에 큰따옴표 `"` | 인용된 필드 안에서 `""` 로 이스케이프 |
| 본문에 개행 `\n` | 인용된 필드 안에 그대로 (멀티라인 셀) |
| 빈 값 | 빈 문자열 (`,,` 연속) |

**예시**:
```csv
common,quote_example,"그가 ""안녕""이라 했다.","He said ""hi"".",인용 처리 예,locked
common,multiline,"첫 줄
두 번째 줄",First line\nSecond,멀티라인 셀,review
```

→ 도구가 자동 escape — 작가는 Excel·Google Sheets 에서 그대로 입력하면 됨.

### 2.4 status 워크플로

```
draft → review → locked
  ↑       ↑        ↑
  작가    PM 검토   배포 활성
```

- `draft`: 작가 작성 중·검수 대기
- `review`: PM 검토 중·QA 가능
- `locked`: 배포 활성·변경 시 changelog 필수

## 3. JSON 포맷 사양 (모듈 SoT)

### 3.1 위치·파일명
```
packages/modules/{moduleId}/i18n/{locale}.json
```
- 모듈 14 × 언어 2 (ko·en) = 28 파일 (현재). 신규 언어 추가 시 자동 확장.

### 3.2 구조 — 평탄 객체 (key·value)
```json
{
  "name": "정원",
  "title": "🌱 성장의 정원",
  "subtitle": "씨앗을 심고 30초 기다리면 식물이 자라요.",
  "action.plant": "씨앗 심기",
  "sample.tooltip": "정원 툴팁 텍스트 (PM 검수 대기)"
}
```

- 중첩 객체 X — `.` 구분 키만 (`useTranslation` 의 namespace 와 정합)
- 들여쓰기: 2 스페이스
- 키 정렬: **유지 순서** (도구가 alphabetical sort 옵션 제공)
- 종결: 파일 끝 개행

### 3.3 namespace 매핑
- 모듈 ID = i18n namespace (예: `zone-garden`)
- 화면 사용: `useTranslation('zone-garden')` + `t('title')`

## 4. 워크플로우 (5 단계)

### 4.1 작가 — CSV 작업
```
1. PM 이 최신 i18n_export.csv 작가에게 전달 (Drive·Slack·Email)
   또는 작가가 직접 `pnpm i18n:export` 실행 후 받기
2. Excel·Google Sheets·번역 도구 (CAT·Trados 등) 열기
3. ko/en 컬럼 편집·status='draft' 유지
4. CSV 저장 (UTF-8, 콤마 구분, RFC 4180)
5. PM 에게 회신
```

### 4.2 PM — 검수·import
```
1. 받은 CSV 를 `packages/i18n-source/i18n.csv` 에 덮어쓰기
2. `pnpm i18n:check` — 자동 검수 (빈 값·길이·영감원·중복)
3. 통과 시 `pnpm i18n:import` — CSV → 모듈 JSON 변환
4. `git diff` 로 변경 확인
5. status='review' 또는 'locked' 갱신
6. `git commit` (의미 있는 메시지)
```

### 4.3 코드 — 자동 반영
- `git pull` → JSON 파일 갱신 → Vite HMR 또는 `pnpm dev` 재시작
- 화면에 즉시 반영 (코드 변경 X)

### 4.4 신규 키 추가 (코드측)
```
1. 화면에서 `t('zone-garden:new_key')` 추가
2. `packages/modules/zone-garden/i18n/ko.json` 에 빈 값으로 `"new_key": ""` 추가
3. `pnpm i18n:export` 실행 — CSV 에 자동 반영 (빈 셀)
4. 작가에게 전달 — 4.1 흐름
```

### 4.5 신규 언어 추가 (Phase 2)
```
1. PM 이 `packages/i18n-source/i18n.csv` 에 컬럼 `ja` (또는 `zh` 등) 추가
2. `pnpm i18n:import` — 모듈마다 `i18n/ja.json` 자동 생성
3. frontend `src/i18n/index.ts` 에 ja 추가
```

## 5. 도구 (4종)

| 도구 | 명령 | 역할 |
|---|---|---|
| **exportCsv** | `pnpm i18n:export` | 모듈 JSON → 통합 CSV (작가 export 용) |
| **importCsv** | `pnpm i18n:import` | 통합 CSV → 모듈 JSON (PM 검수 후) |
| **validate** | `pnpm i18n:check` | CSV/JSON sanity (빈 키·길이·영감원·중복) |
| (Phase 2) **diff** | `pnpm i18n:diff` | import 전 변경 미리보기 |

위치: `tools/i18n/{exportCsv·importCsv·validate}.py` + bash wrapper.

## 6. 검수 게이트 (validate.py)

| # | 검사 | 위반 | 처리 |
|---|---|---|---|
| 1 | 빈 키 (`ko` 또는 `en` 비어있음) | 누락 | 🟡 경고 (draft 통과·locked 거절) |
| 2 | 길이 한계 | ko ≤200자 · en ≤300자 (영어 길이 ↑) | 🟡 경고 |
| 3 | 영감원 식별어 grep | 정국·태민·창빈·레오·카이·... | 🔴 거절 |
| 4 | 중복 key | 같은 (module, key) 다중 행 | 🔴 거절 |
| 5 | 미정의 module | 14 모듈 외 ID | 🔴 거절 |
| 6 | placeholder 일관성 | `{{var}}` 가 ko/en 양쪽에 동일 | 🟡 경고 |
| 7 | HTML 태그 | `<br>·<b>` 등 | 🟡 경고 (의도적 시 OK) |

자동 grep 통과 시 PM 사람 검수로 진행.

## 7. 버전 관리

### 7.1 git 추적 대상
- ✅ `packages/i18n-source/i18n.csv` — CSV 소스
- ✅ `packages/modules/{m}/i18n/{ko,en}.json` — 빌드 산출물 (코드 import)
- ❌ 작가 working file (`*.xlsx`·임시 csv) — `.gitignore`

### 7.2 commit 메시지 컨벤션
```
i18n: <module> <action>

- module: 변경된 모듈 ID 또는 'multi'
- action: add (신규 키) / update (텍스트 변경) / lock (status=locked) / translate (언어 추가)

예:
  i18n: zone-garden add title·subtitle·6 action keys (ko·en)
  i18n: multi update — 작가 1차 검수 회신 (45 키)
  i18n: my-aidong lock — Phase 1.5 발사 전 락
```

### 7.3 changelog
- 각 모듈 i18n 폴더에 `CHANGELOG.md` (Phase 1.5+ 도입 검토)
- 또는 git log 만 (1주차 단순화)

### 7.4 충돌 해결
- CSV 가 SoT — 충돌 시 작가/PM 회의로 결정
- JSON 은 빌드 산출물 — 충돌 시 항상 `pnpm i18n:import` 재실행

## 8. 신규 모듈 i18n 추가 절차

```
1. packages/modules/{newmod}/i18n/ko.json 신설 (빈 객체 {})
2. packages/modules/{newmod}/i18n/en.json 신설 (빈 객체 {})
3. packages/modules/{newmod}/i18n/index.ts 작성 (ModuleI18n shape)
4. packages/modules/{newmod}/package.json `exports['./i18n']` 추가
5. frontend src/i18n/index.ts import 추가
6. `pnpm i18n:export` 실행 — 통합 CSV 에 모듈 행 자동 추가
```

→ Phase 1.5+ : 도구가 신규 모듈 자동 감지 + 빈 JSON 자동 생성 (`pnpm i18n:scaffold`).
